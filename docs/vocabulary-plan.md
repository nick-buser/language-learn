# Vocabulary acquisition — feature plan

*Drafted 2026-06-10. The grammar folios (loom, forge, dial, cabinet) cover structure; this
plan covers the other half of the language: words. It is a roadmap, not a spec — each phase
gets its own design pass when it starts.*

## The shape of the problem

The learner is a beginner in Korean with strong Japanese. The single biggest asymmetry that
creates: **a huge fraction of Korean vocabulary is already known**, latently, as Sino-Japanese.
한자어 (Sino-Korean words) make up ~60% of the Korean lexicon, and the on'yomi ↔ Sino-Korean
sound correspondences are regular enough to be *learnable as rules* rather than word-by-word.
A Japanese speaker who internalizes the sound bridge gets thousands of words at a steep
discount. Vocabulary strategy should exploit this before anything else.

## Phases

### 1. Cognates (now)

The cognate bridge: Sino-Japanese ↔ Sino-Korean correspondence as a working instrument.

- **The sound bridge** — the systematic mappings (Middle Chinese finals surviving differently
  in each language): JP -く/-き ↔ KR -ㄱ, JP -つ/-ち ↔ KR -ㄹ, JP long vowels ↔ KR -ㅇ/-ㅂ,
  JP ん ↔ KR -ㄴ *and* -ㅁ (Korean preserves a distinction Japanese merged). Interactive:
  pick a rule, see it run across specimens; pick a word, see its per-character derivation.
- **The cognate ledger** — a dictionary-shaped browser of exemplar entries: hanja, Korean
  (hangul + RR), Japanese (kanji + kana), gloss, per-character sound-rule derivation,
  cognacy status (true cognate / skewed sense / false friend), specimen sentence.
- Data: `src/data/koreanCognates.js`, hardcoded, 10–20 entries, schema written as the
  expected payload of the future dictionary API. Interface first; bulk seeding later.

### 2. Dictionary system (next)

A real lexicon behind the instruments.

- Backend + DB (likely the homelab stack); entries follow the schema piloted in phase 1,
  generalized beyond cognates (native Korean vocab, loanwords, conjugation class, frequency
  band, source/provenance of each entry).
- Bulk seeding from open sources (KRDICT, Wiktionary extracts, frequency lists) with the
  hand-checking bar applied to anything surfaced in an instrument.
- The frontend instruments stay payload-compatible: swapping hardcoded modules for API calls
  should not require component surgery.

### 3. Known-vocabulary persistence

Track what the learner actually knows.

- Per-word state (unseen / met / learning / known), timestamps, source of acquisition
  (which instrument/article introduced it).
- Starts as localStorage keyed to the same schema; graduates to the backend with the
  dictionary. Every later feature (reading, SRS) reads and writes this state.

*Status 2026-06: piloted, interface-first, in the word bank folios (`#/ko/vocab`,
`#/ja/vocab`). The holdings ledger (`WordLedger`) is the browsing/triage face over a
hardcoded exemplar bank (`koreanVocab.js` / `japaneseVocab.js` — the generalized
VocabEntry schema); `useVocabStore` holds the four-state taxonomy in localStorage
(`atlas.<lang>.vocab.v1`). Still pending from this phase: source-of-acquisition beyond
ledger/review (arrives with reading), backend persistence (arrives with phase 2).*

### 4. Extensive reading

Comprehensible input tuned to the known-vocabulary set.

- Articles at ~95–98% known-word coverage. While the known set is small, articles are
  **autogenerated** (LLM-written to a vocabulary constraint + grammar ceiling) — real corpora
  come later when the known set can support them.
- Unknown-word taps: glossed inline from the dictionary, one tap to mark as learning.
- Reading is also the harvest: words met in context feed the SRS queue with their sentence
  attached.

### 5. Simple SRS

Review, kept deliberately boring.

- A plain interval scheduler (SM-2-ish) over the learning set; cards are dictionary entries
  with their harvested context sentences, hangul + RR + JP bridge per the house conventions.
- No gamification chrome — it's a drawer in the atlas, not an app within the app.

*Status 2026-06: piloted in the same word bank folios (`ReviewDrawer` + `srs.js`):
day-granular SM-2-ish (four grades, one ease factor), interval previews on the grade
buttons, 21-day graduation to known, seven-day due forecast, session receipt. Cards
carry the bank's specimen sentences for now; harvested context sentences arrive with
phase 4.*

## Sequencing notes

- Phase 1 stands alone and is pure frontend — no blockers.
- Phases 3 and 5 jumped the queue (2026-06) as frontend pilots over exemplar banks —
  the word bank folios. This inverts the written order deliberately: the instruments
  define the payloads the phase-2 backend must serve, not the other way round.
- Phases 2–3 land together (the DB serves both); 4 and 5 depend on 3.
- At every phase the **schema is the contract**: instruments built against exemplar data
  modules must survive the swap to the API unchanged.
