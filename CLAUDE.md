# CLAUDE.md — The Polyglot's Atlas

Interactive grammar/vocabulary "instruments" for language study (currently Korean-via-Japanese),
Vite + React, Aburaya design language. Read `README.md` for the full map of folios, instruments,
and file layout before touching anything.

## Git discipline — commit as you go

**Commit each coherent unit of work as soon as it builds, not at the end of the session.**
Work lost between commits is work lost; a session's output should never sit uncommitted while
the next feature starts. Concretely:

- One feature/instrument/refactor = one commit. If a change serves two purposes, split it.
- Commit when the unit is done and `npm run build` passes — then start the next thing.
- Messages carry the paper trail: a title that names the change, a body that records **what**
  was built, **how** it's shaped, and **why** it exists / why this design. Write for the person
  doing archaeology in six months. (See the 2026-06 commits for the bar.)
- Order commits so the tree builds at every step: data/components first (unreferenced files are
  fine), wiring/shell after.
- End commit messages with `Co-Authored-By` per the harness footer convention.
- Push branches and open PRs as part of the normal flow: when a unit is ready for review, push
  the feature branch and open a PR into `main` on the homelab Gitea (`nick-b/polyglot-atlas`,
  remote `gitea`; the `open-pr` skill does both). Work reaches `main` only through a PR — never
  commit or push directly to `main`. (Merging the PR is the user's call.)

## Architecture conventions

- **Data is not components.** Linguistic content (specimens, verbs, particle cards, cognates…)
  lives in `src/data/*.js` modules; instrument mechanics live in `src/components/`. Adding
  content must be a data edit, never component surgery. Document each data module's shape with
  a JSDoc typedef at the top — these schemas are previews of future backend/DB payloads, so
  treat them as contracts.
- **Pages are thin.** `src/pages/*` lays out masthead → lede → instruments → coda marginalia;
  the instruments do the work. New folio = one `LANGS` entry in `App.jsx` + one page file.
- Routing is hash-based: `#/<lang>/<folio>` (e.g. `#/ko/grammar`).
- Styles: one CSS file per instrument family in `src/styles/`; tokens come from `aburaya.css`
  (canonical source: `design_system.zip` → `colors_and_type.css`). Never hardcode colors,
  radii, easings — use the tokens.

## Content conventions (non-negotiable)

- Korean always carries **hangul + Revised Romanization**, romanized to reflect pronunciation
  (합니다 → *hamnida*, liaison marked: 책이 → *chae-gi*). Honor the global `showReadings` toggle.
- Show the **Japanese bridge** wherever the mapping is real — the learner is a beginner in
  Korean with strong Japanese, and transfer is the whole pedagogy. Honor the `showJp` toggle
  (Korean pages only) so bridges can be hidden for self-testing.
- The didactic bar is Tufte: every pixel earns its place by making a grammatical idea *felt*.
  Discovery-driven — instruments invite manipulation, and "lantern note" eureka panels fire
  once per insight per visit. No quizzes-with-chrome; working machines.
- Aburaya guardrails: dark is home; gold is matte and rationed; red is signal, never body
  text; glow not shadow at night; motion drifts and settles, never bounces; no emoji — glyphs
  are real CJK/hangul characters.
- Hand-check all linguistic content (hangul, RR, readings, bridges) before committing; a
  wrong specimen is worse than a missing one.

## Roadmap context

The Korean track is heading toward a **vocabulary acquisition system**: cognate handling
(Sino-Japanese ↔ Sino-Korean), then persistence of known vocabulary, autogenerated extensive
reading, simple SRS, and a real dictionary backend + DB. Until the backend exists, dictionary-
shaped features use hardcoded data modules with 10–20 exemplar entries whose schema mirrors
the expected API payload — interface first, bulk seeding later. See `docs/vocabulary-plan.md`.
